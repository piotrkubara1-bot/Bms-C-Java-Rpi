BMS Host

Projekt sklada sie z backendu Java, Web GUI, desktop viewera JavaFX oraz programu C na Raspberry Pi.

Przeplyw danych

bateria -> BMS -> Raspberry Pi -> tinybms_rpi -> SSH -> Windows Bash -> DataManager.java -> BmsApiServer -> GUI

Program rpi/tinybms_rpi.c czyta TinyBMS przez /dev/ttyUSB0, buduje ramki z CRC MODBUS i moze wypisac dane w formacie akceptowanym przez BmsApiServer.

Najwazniejsze pliki

  • src/main/java/BmsApiServer.java - backend HTTP i parser danych BMS.
  • src/main/java/DataManager.java - odbiera linie ze standardowego wejscia i wysyla je do API.
  • src/main/java/BmsUartSender.java - wariant Java czytajacy UART lokalnie na PC.
  • src/main/resources/dashboard.html - Web GUI.
  • rpi/tinybms_rpi.c - program C dla Raspberry Pi.
  • rpi/stream_to_windows_java.sh - potok RPi do Windows przez SSH.
  • scripts/windows_receive_from_ssh.sh - odbior danych po SSH i start DataManager.
  • docs/RPI_SSH_PIPELINE.md - instrukcja konfiguracji RPi/SSH.

Format danych

BMS,module_id,pack_voltage_v,current_a,soc_raw,status_decimal,cell_1_mv,cell_2_mv,...

Taki wiersz mozna wyslac bezposrednio do endpointu /api/ingest albo do DataManager.java przez standardowe wejscie.

Sprawdzony wariant: laptop jako hotspot

Gdy laptop udostepnia hotspot Windows, laptop zwykle ma adres 192.168.137.1, a Raspberry Pi dostaje adres 192.168.137.x. Na RPi sprawdz to komenda: 

ip route

Adres po default via to adres laptopa. Pelny start potoku z RPi: 

PC_USER=piotrek \
PC_HOST=192.168.137.1 \
REMOTE_COMMAND='cd /d "C:\Users\Piotrek\Desktop\Bms-C-Java-Rpi-main" && bash -lc "bash scripts/windows_receive_from_ssh.sh"' \
./rpi/stream_to_windows_java.sh

Endpointy testowe

/api/ingest przyjmuje dane metoda POST. /api/latest zwraca ostatnie dane metoda GET. Nie wysylaj POST na /api/latest, bo dostaniesz Method not allowed. 

printf "BMS,1,52.100,0.000,75000000,151,3260,3261,3262,3264\n" | ssh piotrek@192.168.137.1 'curl.exe -X POST http://127.0.0.1:8090/api/ingest --data-binary @-'
ssh piotrek@192.168.137.1 'curl.exe http://127.0.0.1:8090/api/latest'

Poprawny ingest zwraca {"accepted":1,"heartbeat":0,"rejected":0}.

Typowe problemy

  • Connection refused - backend API nie dziala albo DataManager trafia w zly adres.
  • Connect timed out - firewall Windows blokuje port albo backend nie nasluchuje na interfejsie sieciowym.
  • Listening on 0.0.0.0:8090 - poprawny log backendu dla pracy przez hotspot/WSL.
  • java command: java.exe - poprawny log odbiornika, gdy SSH uruchamia bash w WSL.
  • UnsupportedClassVersionError - javac i java sa z roznych wersji JDK.
where java
where javac
java -version
javac -version

Pierwsze where java powinno wskazywac aktualny JDK, np. C:\Program Files\Java\jdk-25\bin\java.exe.
